### ⚠️ 异常

#### P1-7:文章分类计数与"已发布 8"不符

**Dashboard 显示:**
- 草稿 1, 已发布 8, 总数 9
- Top 5 分类合计: 4 + 3 + 1 + 1 = **9**

**矛盾点:** "草稿 1 + 已发布 8 = 9" 与 Top 5 合计 9 一致,但其中"产品知识 1"实际是最新草稿「轮毂升级指南」(看 articles 截图)。

**结论:** Top 5 分类统计**未过滤 status**,把草稿也算进去了。

**修复建议:** `getDashboardSummary` 内 SQL 加 `WHERE status = 'published'`。

#### P1-8:本月预约 = 0 但 692 PV

**预期:** 692 pageviews 应有少量预约转化(行业平均 0.1%-1%)。

**实际:** 完全为 0。

**可能原因:**
1. 站点暂无预约入口 → 检查 Footer / Hero / Contact 是否有预约表单
2. 预约事件未埋点 → 检查 `src/lib/analytics.ts` 是否 track `type=booking`
3. 预约提交后未走 track → 检查 /api/bookings 是否调用

**建议:** 用 /admin/analytics 的「事件类型分布」对照:bar 图显示"页面浏览 700 / 点击 5" → **点击事件几乎全无**,说明 analytics `type=click` 也没数据。整站埋点严重不足。

#### P3-3:门店网络"活跃 22 禁用 0"

**问题:** 22 家门店全部"活跃",但截图 stores 列表显示 **22 条全是草稿状态**(见 admin__stores 视觉报告)。

**矛盾:** 后台管理 list 标 `草稿` → 前台 /api/stores 仍返回 → 表语义混乱。

**可能解释:** "status" (草稿/已发布/下架) 与 "active" (布尔) 是两个独立字段:
- `status` = 内容编辑态(草稿→已发布→下架)
- `active` = 营业中/暂停

如果是这样,语义 OK,但 UI 没区分会让 admin 误判 → P2 文案建议。
