## 5. 风险与回滚

### 已知 P2 风险（不阻断）

1. **`Province.code` 唯一索引**：现存 3 条历史 Province 的 `code` 为 NULL，重复 seed 不会冲突；后续可在 `seedRegions` update 子句把 NULL → 实际 code。
2. **`data.ts` 静态 fallback**：若 DB 不可用，`getStores/getProvinces/getCities` 会回退到 `@/lib/store` 的 7 条 mock 门店。建议 Phase 3 移除。
3. **`/api/regions` 无缓存**：每次都查 DB。生产可加 `next: { revalidate: 3600 }` 兜底。

### 回滚方案

```bash
# 1. 找到 P1 之前的最后一个 commit
git log --oneline main | grep -E "0bab323"

# 2. 回滚（不推荐 reset，改 revert）
git revert -m 1 0a48d91 e5111d1 b33cc6f 5088730

# 3. 数据库回滚（如果想彻底）
npx prisma migrate resolve --rolled-back 20260614134645_add_region_code_type
# 手动 DROP COLUMN code, type 在 Province 和 City
```

⚠️ **回滚前必须评估**：
- `prisma/seed.ts` 已改写为 upsert；回滚到旧版会重新出现 `deleteMany()` 风险
- `src/lib/store-regions.ts` 已删除；回滚后 `/api/store-regions` 重新可用，但 `POST /api/stores` 的 `findRegion/findCity` 调用也回来，会再次出现 P2003 风险

**建议**：不回滚，保留新代码，前向修复（如有 P0 缺陷）。
