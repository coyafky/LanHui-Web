## 1. 系统目标

用户行为分析系统不是“统计所有点击”，而是帮助总部回答经营问题：

1. 车主最关注哪些轻改产品？
2. 车主最关注哪些新能源车型专题？
3. 哪些文章带来了进一步浏览或咨询？
4. 哪些门店被查看、拨打电话或咨询得最多？
5. 用户从进入网站到表达咨询意向，经过了哪些关键步骤？
6. 哪些页面有访问但没有进一步兴趣或咨询？

系统应当让运营人员能够据此调整内容、产品展示、专题建设和门店运营。

---

## 2. 一期产品原则

### 2.1 只采集有决策价值的行为

- 自动采集页面浏览。
- 采集产品、车型、文章、门店和咨询等关键行为。
- 不采集鼠标移动、滚动每一像素、普通导航展开等无经营意义行为。
- 不把所有按钮统一记为无法解释的 `click`。

### 2.2 指标口径必须可解释

- PV 只代表页面被浏览，不等于用户感兴趣。
- 点击产品详情、查看门店、打开微信、拨打电话等行为代表不同强度的兴趣。
- 每个指标必须说明事件来源、计算方式和不能代表什么。
- 看板出现零数据时，应区分“真实为零”和“埋点未接入或失效”。

### 2.3 采集失败不影响官网

- 埋点请求失败不能阻止页面跳转、咨询或电话拨打。
- 客户端采集需要缓冲和失败重试。
- 服务端异常只影响数据完整度，不影响公开站核心功能。

### 2.4 一期保持轻量

- 不建设通用 BI 平台。
- 不做实时大屏。
- 不做复杂用户画像、跨设备识别和广告归因。
- 不采集姓名、电话、微信号等个人内容。
- 不在 `/admin` 后台采集运营人员行为作为车主数据。

- 汽车窗膜。

- 问界。
- 小米 SU7。

### 4.3 内容

### 4.4 门店

- 点击门店电话、微信或导航属于更强的咨询/到店意向。

## 5. 用户兴趣层级

```text
访问
  ↓
页面浏览
  ↓
内容/产品兴趣
  ↓
门店兴趣
  ↓
咨询意向
```

| 层级 | 含义 | 示例 |
|---|---|---|
| L1 访问 | 用户进入网站页面 | 首页 page view |
| L2 内容兴趣 | 用户主动进入具体内容 | 产品详情、车型专题、文章详情 |
| L3 门店兴趣 | 用户主动查看服务门店 | 门店详情 |
| L4 咨询意向 | 用户主动尝试联系 | 电话、微信、导航、表单 |

不同层级不能简单相加为同一个“总事件”指标。

## 6. 一期事件字典

### 6.1 事件命名原则

- 使用稳定的业务事件名，不使用 UI 文案作为事件类型。
- 页面按钮改名不能导致历史数据断裂。
- 实体通过独立字段或结构化 metadata 记录。
- 事件名采用小写蛇形命名。

### 6.2 核心事件

| 事件 | 触发条件 | 必需数据 | 业务意义 |
|---|---|---|---|
| `page_view` | 公开站路由完成加载 | pathname、pageType | 流量 |
| `product_view` | 进入具体产品线页面 | productKey、pathname | 产品兴趣 |
| `topic_view` | 进入车型/主题专题 | topicKey、pathname | 车型专题兴趣 |
| `article_view` | 进入已发布文章详情 | articleId、category | 内容兴趣 |
| `store_view` | 进入营业中门店详情 | storeId、province、city、level | 门店兴趣 |
| `contact_click` | 点击微信、电话或其他咨询入口 | channel、sourceArea、relatedEntity | 咨询意向 |
| `navigation_click` | 点击门店导航入口 | storeId | 到店意向 |
| `form_submit_success` | 业务表单服务端确认成功 | formName、relatedEntity | 有效提交 |

### 6.3 暂不作为核心指标的事件

- 普通菜单点击。
- 轮播图左右切换。
- 展开/收起导航。
- 返回顶部。
- 无业务含义的任意按钮点击。
- 表单仅点击“提交”但服务端失败。

必要时可用于交互诊断，但不能进入经营 KPI。

### 6.4 页面类型

`pageType` 使用固定枚举：

```text
home
brand
product_index
product_detail
topic_detail
article_list
article_detail
store_region
store_detail
contact
other
```

### 6.5 产品与专题标识

产品和专题使用稳定 key，不使用中文显示名称作为唯一标识：

```text
productKey:
electric_steps | wheels | chassis | window_film | color_film | ppf

topicKey:
wenjie | xiaomi | zeekr | flooring
```

新增产品或专题时，必须同时更新事件字典和分析展示名称映射。

## 7. 事件公共数据

每条事件至少包含：

| 字段 | 必需 | 说明 |
|---|---:|---|
| eventName | 是 | 稳定业务事件名 |
| pathname | 是 | 发生事件的页面 |
| occurredAt | 服务端生成 | 服务端接收时间 |
| pageType | 是 | 页面类型 |
| sessionId | 推荐 | 匿名会话标识，是否进入一期待确认 |
| entityType | 否 | product/topic/article/store |
| entityId | 否 | 稳定实体 ID 或 key |
| sourceArea | 否 | header/hero/card/footer/detail 等 |
| channel | 否 | phone/wechat/navigation/form |
| metadata | 否 | 白名单内补充字段 |

禁止放入 metadata：

- 用户输入的电话号码。
- 微信号。
- 姓名。
- 完整表单正文。
- Cookie、token 和认证信息。
- 任意未定义的大对象。

## 8. 指标口径

### 8.1 基础指标

| 指标 | 计算方式 | 解释 |
|---|---|---|
| PV | `page_view` 数量 | 页面总浏览次数 |
| 产品兴趣次数 | `product_view` 数量 | 产品页被主动查看次数 |
| 专题兴趣次数 | `topic_view` 数量 | 车型专题被查看次数 |
| 文章阅读次数 | `article_view` 数量 | 文章详情被查看次数 |
| 门店查看次数 | `store_view` 数量 | 门店详情被查看次数 |
| 咨询意向次数 | `contact_click + navigation_click + form_submit_success` | 主动联系或到店意向 |

### 8.2 推荐核心 KPI

一期推荐首页只突出三个经营 KPI：

1. **产品兴趣次数**：用户主动查看产品或专题的次数。
2. **门店兴趣次数**：门店详情查看次数。
3. **咨询意向次数**：电话、微信、导航和有效表单行为。

PV 作为背景流量指标展示，不作为唯一成功指标。

### 8.3 基础转化率

```text
产品兴趣率 = 产品/专题兴趣次数 ÷ PV
门店兴趣率 = 门店查看次数 ÷ 产品/专题兴趣次数
咨询意向率 = 咨询意向次数 ÷ 门店查看次数
```

注意：

- 在没有 `sessionId` 去重时，这些是“事件比率”，不是严格用户转化率。
- 分母为 0 时显示“暂无足够数据”，不能显示 `0%` 或报错。
- 机器人和内部测试流量会影响结果，需要数据质量策略。

## 9. 后台分析页面

### 9.1 页面目标

运营人员进入 `/admin/analytics` 后，应在一屏内看到：

- 当前时间范围。
- 三个核心经营 KPI 和 PV。
- 产品与专题兴趣排行。
- 热门文章和热门门店。
- 咨询渠道分布。
- 数据是否健康。

### 9.2 时间范围

一期支持：

- 最近 7 天。
- 最近 30 天。
- 最近 90 天。

所有卡片和图表必须使用同一个时间范围。切换范围后显示加载状态，失败时保留旧数据并允许重试。

### 9.3 页面模块

#### KPI 区

- PV。
- 产品/专题兴趣次数。
- 门店查看次数。
- 咨询意向次数。

#### 趋势区

- 每日 PV。
- 每日产品/专题兴趣。
- 每日门店查看。
- 每日咨询意向。

允许切换指标，不需要在同一图中堆叠过多折线。

#### 兴趣排行

- 产品线 Top 6。
- 车型专题排行。
- 每项显示次数和占全部兴趣的比例。

#### 内容排行

- 热门文章 Top 10。
- 显示标题、分类、阅读次数和咨询贡献（数据具备时）。

#### 门店排行

- 热门门店 Top 10。
- 显示门店、所在省市、门店等级、查看次数和咨询次数。

#### 咨询渠道

- 微信。
- 电话。
- 导航。
- 表单。

#### 数据健康

- 最近一次成功接收事件时间。
- 各核心事件最近 24 小时数量。
- 未识别事件数量。
- 缺少实体 ID 的事件数量。
- 埋点长期为零的页面或模块提示。

## 10. 筛选与下钻

### 10.1 一期筛选

- 时间范围。
- 产品线。
- 车型专题。
- 文章分类。
- 门店省份。
- 门店城市。
- 门店等级。
- 咨询渠道。

### 10.2 联动规则

- 选择省份后，城市只显示该省份下的城市。
- 选择产品线时，排行和趋势只统计对应产品。
- 清除筛选恢复当前时间范围的全局数据。
- 筛选条件同步到 URL，刷新后能够恢复。
- 无结果时展示筛选摘要和“清除筛选”，不能伪装成系统无数据。

### 10.3 一期不做

- 自定义拖拽报表。
- 任意 SQL 查询。
- 多维透视表。
- CSV/Excel 导出。
- 用户级行为轨迹。

## 11. 数据采集流程

```text
公开站关键行为
   ↓
客户端事件函数
   ↓
缓冲 / 批量发送
   ↓
公开 Track API
   ├─ 数量限制
   ├─ 事件白名单
   ├─ 字段白名单
   ├─ 数据大小限制
   └─ 基础限流
   ↓
AnalyticsEvent
   ↓
统计查询与数据质量检查
   ↓
后台分析页面
```

### 11.1 客户端规则

- 路由变化自动发送 `page_view`。
- 产品、专题、文章和门店详情应发送对应的语义事件。
- 咨询事件必须带 `channel` 和触发区域。
- 同一次组件重复挂载不应产生无意义的重复事件。
- 页面离开时尽量刷新缓冲区。

### 11.2 服务端规则

- 严格验证事件名和字段。
- 单批事件数量和单条 metadata 大小设上限。
- 无效事件不写入，并计入数据健康统计。
- `storeId/articleId` 等实体存在性校验策略待技术设计。
- 接口失败不能把敏感内部错误返回给公开客户端。

---

## 12. 数据质量

### 12.1 必须监测的问题

- 只有 PV，没有兴趣和咨询事件。
- 某个已上线产品长期没有任何事件。
- 事件缺少产品、文章或门店标识。
- 同一页面短时间异常重复上报。
- 测试环境数据进入生产统计。
- 已删除实体在历史排行中显示为未知。
- 日期趋势缺失无数据日期。

### 12.2 测试流量

- E2E 和自动化测试必须能够标记为测试数据或使用隔离环境。
- 生产看板默认排除明确标记的测试事件。
- 不使用手工删除生产事件作为常规测试清理方案。

### 12.3 历史实体

- 历史事件应保留事件发生时的实体 key。
- 门店暂停或终止合作后，历史统计仍可查看。
- 实体无法关联时显示“历史门店/内容已不存在”，不能静默丢失统计。

---

## 13. 隐私与安全

- 不采集表单内容和个人联系方式。
- IP 如用于限流或安全，应与经营分析用途分离。
- 是否长期保存原始 IP 需要单独评估；推荐不在经营看板中使用。
- metadata 使用字段白名单和大小限制。
- 分析查询只对授权总部员工开放。
- 所有日期、筛选和分组参数必须服务端校验。
- 公开采集接口必须限流，防止恶意灌入。

---

## 14. 页面状态与异常恢复

| 状态 | 处理方式 |
|---|---|
| 首次加载 | KPI 和图表骨架屏 |
| 切换时间范围 | 保留旧数据并显示局部加载 |
| 无任何数据 | 说明尚未积累数据，并展示埋点检查入口 |
| 筛选无结果 | 展示筛选摘要和清除按钮 |
| 查询失败 | 保留旧数据、显示错误和重试 |
| 部分模块失败 | 其他模块继续显示，失败模块单独提示 |
| 数据异常为零 | 提示检查埋点健康，不直接解释为无人关注 |
| 无权限 | 返回 403 页面或重定向，不泄露数据 |
| 日期非法 | 阻止查询并恢复默认范围 |

## 15. 与当前实现的差距

当前代码已有：

- 自动页面浏览。
- 客户端缓冲和 Beacon/fetch 发送。
- 服务端事件白名单、批量写入和限流。
- 7/30/90 天分析页面。
- 总事件、事件类型、热门页面、热门门店和趋势。

当前不足：

- 事件类型过宽，`click` 不能直接表达产品兴趣。
- 没有独立 `product_view/topic_view/article_view/contact_click/navigation_click`。
- 热门页面统计混入不同事件，不等于页面浏览排行。
- 趋势统计当前是全部事件，不一定是 PV。
- 没有产品线和车型专题排行。
- 没有咨询渠道分布。
- 没有数据健康模块。
- 没有稳定会话标识，因此不能严谨计算 UV 和用户级漏斗。
- 当前 IP 被写入数据库，隐私和保留策略需要重新定义。

本文是目标需求，不代表上述差距已经实现。

## 16. 测试矩阵

### 16.1 客户端

- 每次公开路由变化只产生预期的 `page_view`。
- 后台路由不采集车主事件。
- 产品、专题、文章、门店详情产生正确语义事件和实体 key。
- 微信、电话、导航产生正确渠道事件。
- 表单失败不产生 `form_submit_success`。
- 缓冲满、定时、页面隐藏和关闭时发送。
- 发送失败不影响原始用户操作。

### 16.2 Track API

- 合法事件批量写入。
- 非法事件名、缺失字段、超大 metadata 和超批量被拒绝或丢弃并计数。
- 限流正确。
- 不保存禁止的敏感字段。
- 测试事件可被标记和排除。
- 数据库失败返回安全错误。

### 16.3 Stats API

- 未登录 401，editor/admin 权限符合最终矩阵。
- 日期和筛选参数校验。
- 每项指标仅统计对应事件。
- 无数据日期补 0。
- 分母为 0 时比率不报错。
- 门店省市、等级筛选正确。
- 已终止门店仍保留历史统计名称或快照。
- 测试数据默认排除。

### 16.4 后台页面

- 7/30/90 天切换。
- 所有筛选及 URL 持久化。
- 加载、空、筛选无结果、错误和部分失败状态。
- 产品、专题、文章、门店和咨询排行。
- 三视口、键盘访问和图表替代文本。
- 数字、百分比、日期和时区展示一致。

### 16.5 业务验收

- 能回答“最近 30 天哪个产品最受关注”。
- 能回答“最近 30 天哪个车型专题最受关注”。
- 能回答“哪些文章带来最多咨询意向”。
- 能回答“哪个省、城市和门店被查看最多”。
- 能回答“微信、电话、导航哪个咨询渠道使用最多”。
- 当核心事件异常为零时，系统能提示埋点问题。

- 实时在线人数。
- 热力图和录屏。
