## 1. 系统目标

加盟商完成加盟后，由总部员工维护其官网门店资料，并准确控制门店从“准备上线”到“营业展示”“暂停合作”和“终止合作”的生命周期。

系统必须保证：

- 从未上线的门店不会意外出现在官网。
- 暂停合作和永久终止合作有明确区别。
- 停止展示不会删除历史资料和操作记录。
- 门店恢复营业时能够恢复原资料。
- 每次状态变化都能追溯到操作者和原因。

## 3. 门店状态机

### 3.1 状态定义

| 状态 | 含义 | 官网可见 | 是否可编辑 |
|---|---|---:|---:|
| `pending` 待发布 | 从未上线，资料准备中 | 否 | 是 |
| `active` 营业中 | 正常合作并对车主展示 | 是 | 是 |
| `suspended` 暂停合作 | 曾上线，当前暂时隐藏 | 否 | 是 |
| `terminated` 终止合作 | 永久结束或录入作废 | 否 | 默认只读 |

### 3.2 允许的状态转换

```text
pending ──publish──> active ──suspend──> suspended
   │                   ▲                    │
   │                   └──── resume ────────┘
   │
   └──terminate──> terminated

suspended ──terminate──> terminated
```

### 3.3 状态动作规则

| 动作 | 前置状态 | 目标状态 | 前置条件 | 官网影响 |
|---|---|---|---|---|
| 保存待发布 | pending | pending | 最低保存字段待确认 | 无 |
| 发布门店 | pending | active | 展示必填字段和图片校验通过 | 立即展示 |
| 暂停合作 | active | suspended | 二次确认并记录原因 | 立即隐藏 |
| 恢复营业 | suspended | active | 重新核对联系方式和资料 | 恢复展示 |
| 终止合作 | pending/suspended | terminated | 高风险确认并记录原因 | 永久隐藏 |

### 3.4 禁止转换

- `active → terminated`：建议必须先暂停，再确认终止，避免误操作；最终规则待确认。
- `terminated → active`：禁止直接恢复。若重新加盟，建议创建新的合作记录或走特殊恢复流程。
- 不使用“未发布”表达已经终止合作的门店。
- 不物理删除有上线历史的门店。

## 5. 门店字段初稿

| 字段 | 待发布保存必填 | 发布必填 | 说明 |
|---|---:|---:|---|
| 门店名称 | 是 | 是 | 需要重复检查 |
| slug | 否，可自动生成 | 是且唯一 | 官网 URL |
| 门店等级 | 否 | 是 | 用于筛选、分组和官网展示；正式枚举待确认 |
| 省份 | 否 | 是 | DB 权威地区 |
| 城市 | 否 | 是 | 必须属于所选省份 |
| 详细地址 | 否 | 是 | 由总部员工手工填写完整地址，可包含区县、街道和门牌 |
| 联系电话 | 否 | 是 | 中国大陆 11 位数字号码 |
| 营业时间 | 否 | 是 | |
| 门店介绍 | 否 | 待确认 | |
| 封面图 | 否 | 待确认 | 一期至少支持一张门店主图，管理方式见 §8 |
| 展示排序 | 否 | 待确认 | 推荐门店或人工排序 |
| 状态原因 | 状态动作时 | 状态动作时 | 暂停/终止原因 |

### 5.1 明确不采集的字段

- 不采集经纬度。
- 不设置独立区/县字段。
- 区县、街道、道路和门牌号统一由总部员工写入“详细地址”。
- 不要求总部员工填写 `tel:` 格式。
- 数据库只保存 11 位电话号码；前台拨号链接可在展示层按需生成，不作为运营输入字段。

### 5.2 电话校验

- 只接受 11 位数字。
- 不接受空格、短横线、括号、`+86` 或 `tel:` 前缀。
- 保存前自动去除输入框首尾空格，但不静默修改中间字符。
- 不符合规则时阻止发布，并在联系电话字段下显示错误。
- 如果未来需要固定电话，应另增字段和需求，不放宽本字段含义。

## 6. 地区数据不是门店自由文本

### 6.1 数据关系

省份和城市是独立的地区主数据，不是随门店任意填写的普通文本字段：

```text
Province 1 ── N City
Province 1 ── N Store
City     1 ── N Store
```

- `Province` 保存省份名称、编码、排序、启用状态和门店数量。
- `City` 必须属于一个省份，保存城市名称、编码、排序、启用状态和门店数量。
- `Store` 只能关联系统中已启用且关系正确的省份和城市。
- 服务端根据省市主数据生成显示名称，不信任客户端自行提交的省市名称。
- 选择省份后，城市选项只显示该省份下的城市。

### 6.2 地区维度能力

门店列表必须支持：

- 按省份筛选。
- 选中省份后按城市筛选。
- 按省份分组查看门店及数量。
- 按城市分组查看门店及数量。
- 显示“某省共多少家门店”“某城市共多少家门店”。
- 省份或城市无门店时，区分“没有门店”和“地区数据未启用”。

### 6.3 地区主数据边界

- 一期使用系统预置的中国省市主数据。
- 普通门店表单不能创建新省份或城市。
- 是否需要独立的“地区管理”页面，后续单独确认。
- 禁用已有门店关联的省市时必须阻止或要求先处理关联门店。

## 7. 门店列表：搜索、筛选、分组与排序

### 7.1 搜索

支持按以下内容模糊搜索：

- 门店名称。
- 详细地址。
- 联系电话。
- slug。

### 7.2 筛选

| 筛选维度 | 选项 |
|---|---|
| 状态 | 全部、待发布、营业中、暂停合作、终止合作 |
| 省份 | 全部省份 + 已启用省份 |
| 城市 | 依赖所选省份；未选省份时默认不可用 |
| 门店等级 | 全部等级 + 正式等级枚举 |
| 图片完整度 | 全部、有封面图、缺封面图 |

多个筛选条件同时生效，筛选条件应同步到 URL 查询参数，刷新和返回页面后能够保留。

### 7.3 分组视图

提供以下查看方式：

- 不分组：标准数据表。
- 按省份分组。
- 按城市分组。
- 按门店等级分组。
- 按合作状态分组。

每个分组显示门店数量，并支持展开/收起。分组只改变展示方式，不修改数据库数据。

### 7.4 排序

一期至少支持：

- 最近更新优先。
- 最早更新优先。
- 最新创建优先。
- 门店名称正序/倒序。
- 省份顺序后按城市顺序。
- 门店等级顺序。

人工“官网展示排序”是否进入一期仍待确认；后台列表排序和官网展示排序必须是两个不同概念。

### 7.5 结果与空状态

- 无任何门店：提示创建第一家门店。
- 有门店但筛选无结果：展示当前筛选摘要和“清除筛选”。
- 地区加载失败：筛选器显示错误和重试，不把失败误显示为空。
- 分组后某组无数据：不创建空分组。

## 8. 门店封面图管理

### 8.1 已知现状

当前代码已具备：

- 点击选择或拖拽上传。
- 接受 JPG、PNG、WebP，最大 5MB。
- 服务端使用 Sharp 转换为 WebP。
- 每家门店固定保存一张主图。
- 支持预览、替换和删除。
- 替换图片会覆盖旧文件。
- 当前使用本地 `public/images/stores/` 存储，不使用 OSS。

### 8.2 一期推荐方案（待确认）

一期只做“直接绑定门店的单张封面图”，不建设独立媒体库：

1. 先创建门店并获得门店 ID。
2. 在门店编辑页上传一张封面图。
3. 服务端校验类型和大小，统一转换为 WebP。
4. 数据库保存该门店当前封面图路径。
5. 后台可预览、替换或删除。
6. 替换时成功写入新图后再清理旧图，避免失败后门店无图。
7. 删除前二次确认；删除后门店回到默认占位图。
8. 上传、替换和删除均写入操作日志。

### 8.3 必须补齐的管理细节

- 封面图推荐宽高比和最低分辨率。
- 上传前客户端预览。
- 上传中进度、禁止重复提交和取消机制。
- 格式错误、文件过大、图片损坏和服务端失败提示。
- 替换失败时继续保留旧图。
- 图片删除后官网缓存刷新。
- 门店终止合作时保留图片还是转入归档存储。
- 无引用图片和临时文件的定期清理。

### 8.4 一期非目标

- 多图环境相册。
- 跨门店复用图片。
- 独立媒体素材库。
- 图片标签、文件夹和全文搜索。
- 在线裁剪、滤镜和水印。
- OSS 或第三方对象存储迁移。

这些能力在明确业务需求后独立进入媒体管理 PRD。

## 9. 官网展示规则

- 仅 `active` 门店进入公开 API 和 `/agent` 页面。
- `pending/suspended/terminated` 对公众返回 404 或不进入列表。
- 后台可查看所有状态。
- 暂停或终止后，官网缓存和静态页面必须按明确策略更新。
- 恢复营业前需重新校验电话、地址和营业信息。

---

## 10. 状态操作交互

### 发布

- 展示发布前检查清单。
- 缺少发布必填字段时禁止发布。
- 发布成功后提供“查看官网门店”入口。

### 暂停合作

- 二次确认。
- 选择或填写暂停原因。
- 明确提示官网将立即隐藏。
- 失败时保持 `active`。

### 恢复营业

- 展示上次暂停时间和原因。
- 要求重新确认关键联系方式。
- 成功后恢复官网展示。

### 终止合作

- 高风险确认，不与普通停用使用相同按钮。
- 必须填写终止原因。
- 明确提示原则上不可直接恢复。
- 保留门店历史数据、图片引用和操作日志。

## 11. 异常与恢复清单

| 场景 | 预期处理 |
|---|---|
| 门店名称疑似重复 | 提醒并允许核对，不静默创建 |
| slug 重复 | 阻止保存并定位字段 |
| 省市不匹配 | 阻止保存，要求重新选择 |
| 图片上传失败 | 保留其他表单内容，单独重试 |
| 发布失败 | 保持 pending |
| 暂停失败 | 保持 active |
| 恢复失败 | 保持 suspended |
| 终止失败 | 保持原状态 |
| 数据被其他人修改 | 阻止静默覆盖并提示刷新/合并 |
| 会话过期 | 重新登录后尽量恢复输入 |

## 12. 数据迁移影响

现有模型只有：

```text
isActive = true / false
```

实施时需要迁移为明确状态，例如：

```text
pending | active | suspended | terminated
```

迁移前必须制定现有数据映射规则：

- `isActive=true` 的真实门店 → `active`
- `isActive=false` 的真实门店 → 不能自动判断是 `suspended` 还是 `terminated`，需人工清单
- 测试门店 → 单独识别，不得误归类为真实终止合作门店

## 14. 开放问题

1. 门店等级是否进入一期，以及四个等级的正式命名。
2. 是否确认一期采用单张封面图直接绑定门店，不建设媒体库和相册。
3. 封面图的推荐比例和最低分辨率。
4. 是否需要人工排序和推荐门店。
5. 是否需要独立地区管理页面。
6. `active → terminated` 是否必须经过 suspended。
7. 终止合作后重新加盟，是恢复原记录还是创建新记录。
8. 待发布门店允许保存到什么最低完整程度。
