## 1. 为什么重新建立这份 PRD

现有后台文档主要围绕已经实现的页面和接口进行说明，没有先完整回答以下产品问题：

- 总部员工每天需要通过后台完成哪些工作？
- 门店、文章和用户行为数据分别如何进入系统？
- 一次操作会经过哪些状态，失败后如何恢复？
- 后台数据最终影响官方网站的哪个页面？
- 哪些能力属于网站，哪些能力属于加盟商小程序？
- 每项功能如何测试，怎样才算真正完成？

本 PRD 的目标不是罗列页面，而是先把后台拆成可以独立设计、实施和验收的小系统，再为每个系统建立完整闭环。

## 2. 产品定位与系统边界

### 2.1 后台定位

蓝辉后台是供总部员工使用的官方网站运营管理系统，负责：

1. 管理官网展示的加盟门店。
2. 管理官网文章和内容资产。
3. 分析车主对产品、车型专题、文章和门店的兴趣。
4. 查看官网运营健康度和总部员工操作记录。

### 2.2 用户边界

| 用户 | 是否登录 `/admin` | 在网站中的职责 |
|---|---:|---|
| 车主 | 否 | 浏览产品、文章和门店，产生访问与咨询行为 |
| 总部 admin | 是 | 管理门店、文章、数据和后台账号 |
| 总部 editor | 是 | 编辑和发布内容，查看授权范围内的数据 |
| 加盟商 | 否 | 仅作为官网门店信息被展示 |

### 2.3 不属于本系统

- 加盟申请与审核流程。
- 加盟合同、费用、权益和返佣。
- 加盟商进货、客资和收益账户。
- 加盟商后台或加盟商自助编辑门店。
- 产品目录、SKU、库存、价格和车型适配 CMS。
- 小程序功能实现。

以上加盟相关经营能力由蓝辉小程序承载。

## 3. 后台系统全景

```text
蓝辉总部后台
├── 认证与权限
├── 工作台
├── 门店管理系统
├── 用户行为分析系统
├── 文章内容管理系统
├── 图片与媒体能力
├── 操作审计
└── 总部账号管理（一期范围待确认）
```

### 3.1 三个核心业务系统

| 系统 | 业务目标 | 最终影响 |
|---|---|---|
| 门店管理 | 加盟完成后，由总部将门店发布到官网 | `/agent` 门店网络 |
| 用户行为分析 | 判断车主实际关注的产品、车型、文章和门店 | 运营选题、页面优化、咨询转化 |
| 文章内容管理 | 生产和维护官网内容，并支持公众号运营复用 | `/news` 资讯内容与公众号素材 |

### 3.2 共同基础能力

- 登录、角色权限与会话管理。
- 统一导航、页面标题、搜索、筛选、分页和表单。
- 统一加载、空数据、失败、无权限、冲突和成功反馈。
- 图片上传、压缩、引用和失败恢复。
- 操作日志、危险操作确认与可追溯性。
- 单元测试、API 测试、E2E、响应式和可访问性验收。

## 4. 门店管理系统

### 4.1 业务目标

当加盟商在线下或小程序完成加盟后，总部员工可以把门店资料加入官方网站；车主能够按地区找到真实、有效且正在营业的蓝辉加盟门店。

### 4.2 核心用户任务

1. 新加盟门店资料录入。
2. 补充地址、电话、营业时间、介绍和门店图片。
3. 检查官网展示效果。
4. 正式上线门店。
5. 修改门店资料。
6. 暂停展示或恢复营业。
7. 查找某省、市或某个名称的门店。

### 4.3 已确认状态流

```text
待发布 ──发布──> 营业中 ──暂停合作──> 暂停合作
  │                ▲                    │
  │                └────恢复营业────────┘
  │
  └──终止录入──> 终止合作

暂停合作 ──确认不再合作──> 终止合作
```

- `待发布`：门店资料正在准备，从未在官网展示。
- `营业中`：官网可见，可以被车主查询和联系。
- `暂停合作`：曾经上线，当前暂时隐藏，允许恢复。
- `终止合作`：合作永久结束或录入作废，保留历史记录，原则上不可直接恢复。
- 明确区分“从未发布”和“发布后暂停/终止”。
- 现有 `isActive` 布尔字段不足以表达这些状态，实施时需要迁移为明确的门店状态字段。

### 4.4 一期候选功能

- 门店列表、搜索、地区筛选、状态筛选和分页。
- 创建、编辑、预览、上线、停用和恢复。
- 门店封面图（一期多图相册待后续评估）。
- 省、市、详细地址、联系电话和营业时间。
- 官网门店详情链接。
- 重复门店、重复 slug 和资料缺失检查。
- 操作日志。

### 4.5 待澄清

- 旗舰店、尊享店、专营店、会员店等门店等级的正式名称和枚举。
- 是否需要人工排序或推荐门店。
- 是否最终确认一期只做一张封面图。

## 5. 用户行为分析系统

### 5.1 业务目标

帮助总部回答：

- 车主最关注哪条产品线？
- 哪个新能源车型专题更受欢迎？
- 哪些文章真正带来了进一步浏览或咨询？
- 哪些门店被查看和联系得最多？
- 用户从进入官网到咨询，中间经过了哪些页面？

### 5.2 指标分层

#### 流量指标

- 页面浏览量 PV。
- 独立访客或匿名会话数。
- 页面入口和访问趋势。

#### 兴趣指标

- 产品卡片点击。
- 车型专题点击。
- 文章详情查看。
- 门店详情查看。
- 图片或方案展开。

#### 转化指标

- 电话点击。
- 微信咨询。
- 导航到店。
- 表单提交或预约（若未来存在）。

### 5.3 关键原则

- 不把所有普通点击都当作有效事件。
- PV 与明确兴趣行为分开统计。
- 每个埋点必须说明业务意义、触发条件和数据字段。
- 看板必须能够解释数据，不只展示数字。
- 数据采集失败不能阻断用户浏览。
- 不采集不必要的个人敏感信息。

### 5.4 一期候选看板

- 总访问趋势。
- 产品线兴趣排行。
- 车型专题兴趣排行。
- 热门文章。
- 热门门店。
- 咨询行为趋势。
- 页面到咨询的基础路径。

### 5.5 待澄清

- 一期最重要的三个经营指标。
- 是否需要 UV、来源渠道和地区分析。
- 是否接入第三方分析工具，还是仅使用自建埋点。
- 数据保留周期和隐私告知方式。

## 6. 文章内容管理系统

### 6.1 业务目标

让总部运营人员能够稳定地生产、预览和发布官网文章，并减少官网内容与微信公众号运营之间的重复劳动。

### 6.2 文章内容结构

- 标题。
- URL slug。
- 摘要。
- 正文。
- 分类。
- 封面图。
- 正文配图。
- 作者。
- 发布时间。
- SEO 标题和描述。
- 发布状态。

### 6.3 已确认状态流

```text
草稿 ──发布──> 已发布 ──撤回──> 已撤回
  │              ▲                 │
  │              └────重新发布─────┘
  │
  └──归档──> 已归档

已撤回 ──归档──> 已归档
已归档 ──恢复编辑──> 草稿
```

- `草稿`：从未正式发布，可持续编辑和预览。
- `已发布`：官网可见。
- `已撤回`：曾经发布，当前下线，可继续修改并重新发布。
- `已归档`：不再维护，默认只读；恢复后回到草稿。
- 明确区分“从未发布”和“发布后撤回”。
- 是否增加“待审核”状态，仍取决于总部内容审核流程。

### 6.4 正文与图片能力

- 正文标题、段落、列表、引用和链接。
- 上传封面图。
- 上传并插入正文配图。
- 图片压缩、尺寸限制、格式校验和替代文本。
- 上传失败重试。
- 更换图片时检查旧图片引用。
- 删除文章时处理关联图片。
- 官网文章预览。

### 6.5 与微信公众号的关系

一期推荐：

- 后台 CMS 作为网站文章的内容源。
- 提供适合微信公众号的复制或导出能力。
- 不做公众号与网站的双向自动同步。

原因：公众号涉及授权、素材库、富文本格式、草稿箱和发布权限；直接做自动同步会显著扩大一期范围。

最终方案仍需产品确认。

### 6.6 待澄清

- 文章是在后台首发，还是公众号首发后导入网站。
- 是否需要多人审核。
- 编辑器采用 Markdown、结构化富文本还是基础富文本。
- 是否需要定时发布。
- 是否需要文章版本历史和恢复。

## 7. 工作台

工作台不是简单堆放统计卡片，而是帮助总部员工开始当天工作。

一期候选内容：

- 待完善或异常门店。
- 草稿、待发布和近期发布文章。
- 今日/本周访问与咨询变化。
- 热门产品、文章和门店摘要。
- 最近后台操作。
- 新建文章、新建门店等快捷入口。

不展示无法驱动行动的装饰性指标。

## 9. 全局页面状态

每个模块 PRD 和每个页面必须明确以下状态：

| 状态 | 必须回答的问题 |
|---|---|
| 加载中 | 显示什么？能否重复操作？ |
| 空数据 | 是没有数据，还是筛选无结果？ |
| 保存中 | 如何防止重复提交？ |
| 保存成功 | 返回列表、留在当前页还是进入预览？ |
| 字段错误 | 错误显示在哪里？如何聚焦到错误字段？ |
| 业务冲突 | slug 重复、数据已修改时怎么办？ |
| 网络失败 | 用户输入是否保留？如何重试？ |
| 会话过期 | 如何提醒并重新登录？ |
| 无权限 | 隐藏操作还是显示说明？ |
| 危险操作 | 如何确认、撤销或恢复？ |

## 11. 测试闭环原则

每项需求必须绑定至少一个验收用例：

1. 正常主流程。
2. 必填字段和边界值。
3. 网络与服务端错误。
4. 未登录和角色越权。
5. 重复提交和数据冲突。
6. 状态转换及其官网影响。
7. 桌面、平板和移动端。
8. 键盘操作和基础可访问性。
9. 相关旧功能回归。

实施阶段的基础质量门：

```bash
npm run lint
npm run typecheck
npm test
npm run build
npm run test:e2e
```

`typecheck` 结果需要区分项目已知的 9 个测试文件错误和本次新增回归。

## 12. 当前确认与开放问题

### 12.1 已确认

- 后台仅供总部员工使用。
- 加盟商不登录网站后台。
- 加盟商经营功能在小程序中完成。
- 官网需要展示加盟门店。
- 产品线固定，不建设产品 CMS。
- 后台需要门店、用户行为分析和文章内容三个核心系统。
- 先完成需求和 PRD，再编码。
- 文章采用 `草稿 / 已发布 / 已撤回 / 已归档` 状态机。
- 门店采用 `待发布 / 营业中 / 暂停合作 / 终止合作` 状态机。
- 两个系统均明确区分“从未发布”和“发布后下线”。
- 门店不采集经纬度和独立区县，完整地址由总部手工填写。
- 门店联系电话采用 11 位数字规则，不要求运营输入 `tel:` 格式。
- 省份和城市采用独立主数据及关联关系，并作为门店筛选、分组和统计维度。
- 门店列表需要搜索、状态/省/市/等级筛选、分组和排序。

### 12.2 下一步逐项确认

1. 文章与微信公众号的内容源关系。
2. 文章编辑器和图片能力。
3. 门店等级、地图和图片结构。
4. 埋点事件字典与核心经营指标。
5. 内容发布审核与角色权限。
6. 删除、下线、归档和数据恢复规则。
