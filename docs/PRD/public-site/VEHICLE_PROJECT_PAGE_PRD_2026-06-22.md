## 1. “项目页”的定义

本 PRD 暂将“项目页”定义为：

> 围绕一个新能源品牌、车型系列或完整升级主题组织的方案页，例如小米、问界、极氪车型轻改专题，以及地板总成专题。

它不是单个产品详情，也不等于已经完成施工的真实客户案例。

## 2. 页面使命

让车主先选择自己的车型，再看到该车型可考虑的升级方向、产品组合、适配提示和下一步门店确认。

页面不能只是把 Excel 产品图片搬成网格。

## 3. 标准结构

```text
面包屑
→ Hero：车型范围、方案定位、代表图
→ 车型锚点或选择器
→ 该车型常见需求
→ 推荐升级组合
→ 按场景或分类展示产品
→ 适配、图片和商标声明
→ 服务流程
→ 相关产品知识与文章
→ 查找门店 / 咨询
```

## 4. 车型选择

- 桌面端可使用锚点导航或 Tabs。
- 移动端使用横向可滚动标签或选择器。
- 当前车型状态同步到锚点或 URL 时，不创建大量重复 SEO 页面。
- 每个车型显示实际产品数量，不使用虚构统计。

## 5. 推荐升级组合

项目页优先按使用场景组织，而不是直接平铺 SKU：

- 家庭便利。
- 座舱整洁与质感。
- 尾箱和收纳。
- 外观与保护。
- 上下车与日常使用。

组合仅用于帮助理解方向，不构成固定套餐、价格承诺或全部车型可装承诺。

## 6. 产品展示

卡片包括：

- 真实或已确认图片。
- 产品名称。
- 适配车型。
- 产品分类。
- 一句话场景价值。
- 图片状态。
- 查看相关产品知识或咨询。

完整产品表格作为辅助核对区，不应承担页面主要说服任务。

## 7. 适配边界

- 必须提示车型、年款、配置和座椅布局可能影响适配。
- 页面展示不代表汽车品牌官方授权。
- 实际安装以门店确认和车辆状态为准。
- 不承诺所有展示件都有即时库存。

## 8. 图片标准

- 正式产品图统一 4:3。
- 使用固定尺寸元数据和 Next/Image。
- `matched`、`pending-review`、`missing` 三状态继续保留。
- 待复核和缺图项不得用其他车型图片代替。
- 同一专题避免图片背景、比例和视觉重量严重不一致。

## 9. 现有专题调整方向

### 小米、问界、极氪

- 保留车型分组和现有静态数据。
- 在产品网格前增加车型需求和推荐组合。
- 减少“款式数量”作为主要卖点。
- 强化相关产品知识、服务流程和门店闭环。

### 地板专题

- 保留按品牌车型和颜色展示。
- 明确它是“产品主题项目”，不是汽车品牌项目。
- 强化座椅布局、滑轨、尾箱和颜色适配说明。

## 10. 项目页与案例页边界

- 项目页：可规划、可选择的车型或主题方案。
- 案例页：某辆真实车辆已完成的施工记录。

当前没有经过授权的真实客户案例库，因此一期不新增 `/cases`。未来如建设案例系统，应独立定义车辆隐私、图片授权、施工门店和完工信息。

## 11. 埋点

- `product_view`：进入项目页，metadata 标记 `contentType: vehicle_project`。
- `contact_click`：咨询具体车型或产品。
- 页面内车型锚点切换不单独创建大量事件，可写入统一交互 metadata。

## 12. 验收

- [ ] 项目页以车型需求和升级组合为主，不只是产品货架。
- [ ] 用户可以快速定位自己的车型。
- [ ] 每个产品有明确适配和图片状态。
- [ ] 项目页与产品知识双向关联。
- [ ] 页面明确商标、适配和库存边界。
- [ ] 不把示意场景冒充真实客户案例。

## 13. 待确认

“项目页”是否还包括真实完工车辆案例。如果包括，需要另立案例系统 PRD，而不能与车型专题共用同一内容模型。
