# 首页电话咨询转化闭环 PRD

> 面向 Claude Code 架构师、coder、测试使用的执行规范文档。本轮只定义需求、实现边界与验证标准，不要求 Codex 编写功能代码。

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 产品 | 蓝辉轻改 LANHUI 官网 |
| 需求名称 | 首页电话咨询转化闭环 |
| 版本 | v1.0 |
| 日期 | 2026-06-12 |
| 循环阶段 | Discover -> Plan -> Execute -> Verify -> Refine |
| 本轮周期 | 1 天内打穿一个小闭环 |
| 核心转化动作 | 首页电话咨询 |

## 2. 背景与目标

蓝辉轻改官网当前首页已有产品入口与门店入口，但首屏没有明确的电话咨询主转化动作。用户在产生兴趣后，需要更短路径联系品牌方；品牌负责人也需要知道首页电话 CTA 是否被点击。

本轮目标是在首页建立一个最小可验证转化闭环：

1. 访客进入首页。
2. 访客在首屏看到明确的电话咨询 CTA。
3. 访客点击 CTA 后触发系统拨号行为。
4. 系统记录一次首页电话咨询点击事件。
5. 测试可验证 UI、链接、埋点和降级状态。

## 3. 用户与业务结果

### 3.1 用户角色

| 用户 | 需求 |
| --- | --- |
| 车主/潜客 | 快速联系蓝辉，询问产品、价格、施工与到店事项。 |
| 门店/运营人员 | 首页展示的号码可信、统一、可维护，不需要到组件里找硬编码。 |
| 品牌负责人 | 首页电话 CTA 能体现品牌主咨询入口，并可通过数据判断转化意图。 |

### 3.2 本轮业务结果

本轮只追求一个结果：提升首页电话咨询意图的可见性与可追踪性。

本轮不承诺提升真实成交量，因为成交量还依赖真实号码、客服响应、产品价格、门店承接与后续 CRM。

## 4. 范围定义

### 4.1 本轮必须完成

- 首页首屏增加一个明确的电话咨询主 CTA。
- 首页电话 CTA 使用品牌总机电话。
- 门店页继续使用门店电话，不被本轮修改。
- 电话号码来自统一数据源，组件内不得硬编码展示号码或 `tel:` 链接。
- 点击 CTA 时触发 `tel:` 拨号链接。
- 点击 CTA 时记录 analytics click 事件。
- 当品牌总机电话未配置或不可拨打时，首页不得展示假号码，不得生成无效 `tel:` 链接。
- 提供桌面端与移动端可验证的验收路径。

### 4.2 本轮不做

- 不做微信咨询二维码。
- 不做预约表单与留资后台。
- 不做地图导航。
- 不改门店详情页电话逻辑。
- 不重构全站 Header/Footer。
- 不改后台数据模型，除非架构师判断现有数据层无法满足统一读取。
- 不引入第三方统计平台。
- 不做 A/B 测试。

## 5. 电话号码策略

### 5.1 号码类型

系统必须支持两类电话号码：

| 类型 | 使用位置 | 说明 |
| --- | --- | --- |
| 品牌总机电话 | 首页 `/` 的电话咨询 CTA | 代表蓝辉品牌的统一咨询入口。 |
| 门店电话 | 门店详情页、门店列表相关入口 | 代表具体门店的本地服务电话。 |

### 5.2 首页号码规则

首页电话咨询 CTA 必须使用品牌总机电话，不使用任意单一门店电话作为首页默认号码。

品牌总机电话应从品牌数据源读取。推荐字段语义如下：

| 字段语义 | 说明 |
| --- | --- |
| `phone` | 展示给用户看的品牌总机号码。 |
| `phoneTel` | 用于 `href` 的拨号链接，格式必须以 `tel:` 开头。 |

如果当前代码中字段名不同，架构师可以在不破坏现有调用方的前提下调整，但必须保持“展示号码”和“拨号链接”分离。

### 5.3 未配置号码的降级规则

当品牌总机电话未配置、为空、仍为占位文案、或 `phoneTel` 不是有效 `tel:` 链接时：

- 不展示可点击电话 CTA。
- 不展示虚构电话号码。
- 首页可展示一个不可拨号的咨询状态，例如“电话咨询即将开放”。
- 保留现有“浏览产品”“查看门店”等入口，避免首页首屏无行动路径。
- analytics 不记录 `homepage_phone_click`，因为用户没有完成电话点击。

### 5.4 生产发布门禁

生产发布前，品牌负责人必须提供可公开展示和可拨打的品牌总机电话。没有品牌总机电话时，本功能可以进入开发环境或预览环境验证，但不得作为“电话咨询转化已上线”对外宣称。

## 6. 首页体验要求

### 6.1 CTA 层级

首页首屏 CTA 层级应调整为：

1. 主 CTA：电话咨询。
2. 次 CTA：保留现有“浏览产品”和“查看门店”入口，但视觉优先级低于电话咨询。
3. 其他入口保持现有页面信息架构，不抢主 CTA 视觉优先级。

### 6.2 文案要求

主 CTA 文案必须明确表达电话动作。可使用以下方向：

- “电话咨询”
- “立即电话咨询”
- “拨打品牌总机”

如果展示号码，号码应靠近 CTA 或出现在 CTA 内部，便于用户确认点击后会拨打哪里。

禁止使用含糊文案作为主 CTA，例如“了解更多”“马上开始”“获取方案”，因为这些不能明确表达电话行为。

### 6.3 视觉与响应式

- 桌面端首屏应在不滚动或轻微滚动内看到电话咨询 CTA。
- 移动端首屏必须能看到电话咨询 CTA，按钮宽度、行高和文字不得溢出。
- CTA 触控区域高度建议不小于 44px。
- 按钮应有清晰 hover/focus 状态。
- 键盘聚焦时必须可见。
- 不允许文字与图标重叠。

### 6.4 可访问性

- 电话 CTA 必须是语义化链接 `<a href="tel:...">` 或 Next.js `Link` 渲染出的有效链接。
- 可访问名称必须包含电话咨询意图，例如 `aria-label="电话咨询蓝辉轻改"`。
- 如果展示号码，屏幕阅读器读到的信息应能让用户理解拨号对象。

## 7. Analytics 事件规范

### 7.1 事件类型

使用现有 analytics click 能力，不新增第三方依赖。

推荐调用语义：

```ts
trackClick("homepage_phone_click", metadata)
```

### 7.2 事件元数据

`homepage_phone_click` 事件至少包含：

| 字段 | 类型 | 示例 | 说明 |
| --- | --- | --- | --- |
| `target` | string | `homepage_phone_click` | 点击目标。 |
| `source` | string | `hero_primary_cta` | CTA 出现位置。 |
| `phoneType` | string | `brand_main` | 电话类型，本轮固定为品牌总机。 |
| `pathname` | string | `/` | 当前路径，由 analytics SDK 自动或显式记录。 |

不得在埋点 metadata 中发送用户手机号、车牌号、姓名等个人信息。本轮也不采集访客个人信息。

### 7.3 事件触发时机

点击电话 CTA 时立即触发 analytics click 事件，同时保留 `tel:` 默认跳转行为。

实现应注意：如果 analytics 需要异步发送，不得阻塞用户拨号。可使用现有 buffer/sendBeacon 机制。

## 8. 架构师执行规范

架构师负责把需求落成可维护的技术边界，产出给 coder 的执行说明。

必须确认：

- 品牌总机电话的数据源位置。
- 首页组件读取品牌总机电话的方式。
- 是否需要新增小型 `PhoneCta` 或 `ConversionCta` 组件。
- CTA 组件的 props 边界，至少能表达展示号码、拨号链接、点击来源和禁用状态。
- analytics 事件命名、metadata 字段和触发位置。
- 未配置号码时的降级策略。
- 不影响门店电话逻辑。

架构师不得扩大本轮范围到微信、预约、导航、CRM 或后台管理。

## 9. Coder 执行规范

Coder 根据架构师方案实现，必须遵守：

- 不在首页 JSX 中硬编码电话号码。
- 不在多个组件中复制电话 CTA 逻辑。
- 不破坏现有首页产品入口与门店入口。
- 不引入新的 UI 库。
- 使用现有 Tailwind v4、Lucide React、项目组件风格。
- 只在必要文件中修改，避免顺手重构。
- 若新增客户端组件用于点击埋点，该组件范围应尽量小，避免把整个首页转成 Client Component。
- 代码完成后运行项目既有质量命令，至少覆盖 lint、typecheck、build。

## 10. 测试执行规范

测试负责从用户行为与工程质量两条线验证。

### 10.1 功能测试

| 用例 | 步骤 | 期望 |
| --- | --- | --- |
| 首页展示电话 CTA | 打开 `/` | 首屏可见明确电话咨询 CTA。 |
| 电话链接有效 | 检查 CTA `href` | `href` 以 `tel:` 开头且不是占位值。 |
| 点击触发拨号 | 点击 CTA | 浏览器尝试打开拨号链接或保留可验证的 `tel:` 行为。 |
| 埋点触发 | 点击 CTA 并观察 `/api/analytics/track` | 请求或 sendBeacon payload 包含 `homepage_phone_click`。 |
| 无号码降级 | 将品牌总机置为空或占位状态 | 不生成可点击 `tel:`，不展示假号码。 |

### 10.2 响应式测试

至少验证：

- 390px 移动端宽度。
- 768px 平板宽度。
- 1440px 桌面宽度。

每个视口检查：

- CTA 可见。
- 文案不换行到难以阅读的状态。
- 按钮不遮挡其他内容。
- 首屏主要信息仍然清晰。

### 10.3 回归测试

必须确认：

- `/product` 入口仍可访问。
- `/agent` 入口仍可访问。
- Header 与 Footer 没有明显布局破坏。
- 门店页电话不被首页品牌总机覆盖。
- `npm run lint`、`npm run typecheck`、`npm run build` 通过。

## 11. 验收标准

本轮完成必须同时满足：

- 首页首屏出现明确电话咨询主 CTA。
- 首页 CTA 使用品牌总机电话。
- 门店页仍使用门店电话。
- 电话展示值和 `tel:` 链接来自统一数据源。
- 点击电话 CTA 后触发有效拨号链接。
- 点击电话 CTA 后记录 `homepage_phone_click`。
- 品牌总机缺失时不展示假号码，不生成无效拨号链接。
- 桌面、平板、移动端显示正常。
- lint、typecheck、build 通过。
- 测试提交验证记录，包含至少一张桌面截图和一张移动端截图。

## 12. 风险与约束

| 风险 | 约束/处理 |
| --- | --- |
| 品牌总机电话未提供 | 使用降级状态；生产发布前必须由品牌负责人补齐。 |
| 点击埋点与 `tel:` 跳转竞争 | 不阻塞拨号，使用现有 sendBeacon/buffer 能力。 |
| 首页 CTA 过多导致主次不清 | 电话咨询为主 CTA，其他入口降为次级。 |
| 门店电话与品牌电话混用 | 数据层明确区分两类电话，首页只读品牌总机。 |
| 为了埋点扩大 Client Component 范围 | 只创建必要的小客户端交互组件。 |
