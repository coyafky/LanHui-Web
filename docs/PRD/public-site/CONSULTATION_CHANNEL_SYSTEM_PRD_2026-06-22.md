# 蓝辉微信与企业微信咨询承接系统 PRD

> **系统范围**：公开站咨询入口、二维码弹窗、联系页、门店详情、产品与内容 CTA  
> **关联后台**：总部 `/admin` 咨询渠道管理  
> **版本**：v0.1  
> **创建日期**：2026-06-22  
> **状态**：规划中，未授权编码

## 1. 系统目标

把当前硬编码的企业微信二维码弹窗升级为统一的客户咨询承接系统，使车主在首页、产品、车型项目、文章、门店和联系页面都能看到一致、有效、可追踪的咨询入口。

系统需要保证：

- 官网展示的二维码真实有效，不出现“待补充”账号。
- 总部可以在后台替换、启用和停用二维码，无需修改页面代码。
- 页面能够说明用户正在咨询什么，并把来源上下文带入埋点。
- 二维码缺失、失效或图片加载失败时有电话、门店或联系页降级。
- 不把用户微信号、手机号或聊天内容写入行为分析。

## 2. 一期支持的渠道

| 渠道 | 稳定编码 | 说明 |
|---|---|---|
| 企业微信 | `wecom` | 推荐的主要咨询承接渠道 |
| 个人微信 | `wechat` | 业务确认需要时作为补充渠道 |
| 电话 | `phone` | 二维码不可用或用户偏好电话时降级 |

一期只负责展示和引导扫码，不接入企业微信会话 API、客户 CRM、自动分配客服或聊天记录。

## 3. 咨询上下文

每次打开咨询入口都应知道用户从哪里来：

| 上下文类型 | 示例 |
|---|---|
| 全站 | Header、Footer、联系页 |
| 产品 | 电动踏板、窗膜、隐形车衣 |
| 车型项目 | 小米、问界、极氪、地板专题 |
| 文章 | 某篇产品知识或车型方案文章 |
| 门店 | 顺德大良店或未来加盟门店 |

上下文用于显示文案和匿名埋点，不自动把个人信息发送给微信。

## 4. 页面入口改造

### 4.1 Header

- 保留一个紧凑的“微信咨询”入口。
- 点击打开统一咨询弹窗。
- 使用总部默认企业微信。
- 不在 Header 直接展示二维码。

### 4.2 Landing Page

- Hero 不把扫码咨询作为尚未理解品牌时的唯一主 CTA。
- 结尾 CTA 和真实门店区提供微信咨询。
- 弹窗说明“咨询车型与产品方案”。

### 4.3 产品详情

- 页面底部或关键选择区提供“咨询此产品”。
- 传入 `entityType=product` 和稳定 `entityKey`。
- 弹窗标题可显示“咨询电动踏板方案”等上下文。

### 4.4 车型项目

- 提供“咨询此车型方案”。
- 传入 `entityType=vehicle_project`、项目 key 和可选车型 key。
- 不为每张产品卡创建不同二维码资产。

### 4.5 文章详情

- 正文结束后提供“咨询文章相关方案”。
- 传入文章 ID/slug。
- 普通品牌动态文章可以只展示“查看产品”和“查找门店”，不强制出现微信 CTA。

### 4.6 门店详情

- 电话与微信并列。
- 若门店配置了独立有效二维码，则优先使用门店渠道。
- 否则回退总部默认企业微信。
- 弹窗明确当前承接主体，避免用户误以为正在联系具体门店。

### 4.7 联系页

- 以企业微信二维码和真实电话作为主要联系方式。
- 桌面端可直接展示二维码卡片；移动端仍支持点击放大。
- 不展示假 400 电话或占位微信号。

### 4.8 Footer

- 只放“微信咨询”文字入口，不长期显示小尺寸不可扫描二维码。

## 5. 统一咨询弹窗

### 5.1 内容

- 承接主体名称。
- 渠道类型：企业微信或微信。
- 场景化标题。
- 一句扫码后能获得什么帮助。
- 二维码图片。
- 可选账号说明。
- “打开微信扫一扫”提示。
- 电话或联系页降级入口。

### 5.2 状态

| 状态 | 页面处理 |
|---|---|
| 加载中 | 保留弹窗尺寸，显示加载状态 |
| 有效渠道 | 显示二维码 |
| 无可用渠道 | 不显示空白二维码，展示电话/联系页 |
| 图片加载失败 | 显示重试与降级入口 |
| 渠道刚被停用 | 下次解析立即不再返回 |
| 多次点击 | 不重复创建多个弹窗 |

### 5.3 可访问性

- `role=dialog`、`aria-modal=true`。
- 打开后焦点进入弹窗，关闭后返回触发按钮。
- 支持 ESC、关闭按钮和遮罩关闭。
- 弹窗内部焦点循环。
- 二维码有描述性 alt。

## 6. 渠道选择规则

推荐解析顺序：

```text
有效的门店专属渠道
→ 有效的总部默认企业微信
→ 有效的总部个人微信
→ 门店真实电话
→ 联系页
```

渠道必须同时满足：

- `status=active`。
- 二维码图片存在。
- 当前时间在可选生效区间内。
- 所属门店仍为 active（门店专属渠道）。

## 7. 埋点

统一使用 `contact_click`：

| 字段 | 示例 |
|---|---|
| `channel` | `wecom`、`wechat`、`phone` |
| `sourceArea` | `header`、`hero`、`product_cta`、`article_end`、`store_detail` |
| `entityType` | `product`、`vehicle_project`、`article`、`store` |
| `entityId` | 稳定 ID 或 key |
| `contactChannelId` | 实际解析出的渠道 ID |
| `fallbackUsed` | 是否发生降级 |

触发时机：

- 微信/企业微信：用户主动打开包含有效二维码的弹窗时记录一次。
- 电话：用户点击 `tel:` 时记录。
- 只看到按钮或二维码曝光不计为咨询意向。

禁止采集二维码内容、微信号、电话号码和聊天信息。

## 8. 缓存与更新

- 渠道配置属于低频变更数据，可短时缓存。
- 后台启用、停用或替换二维码后必须主动刷新公开缓存。
- 停用渠道不能继续被旧页面长期展示。
- 无数据库构建时使用经过确认的静态总部渠道；若没有真实渠道，则显示联系页降级，不能使用假二维码。

## 9. 当前实现差距

1. `WeChatConsultModal` 固定读取 `/images/brand/wechat-qr.png`。
2. 弹窗固定显示“微信号：fkycoya（待补充）”。
3. `openWeChatModal()` 只能传开关，不能传来源、产品、文章或门店上下文。
4. 当前 Analytics 只允许通用 `click`，没有落地已规划的 `contact_click`。
5. 没有咨询渠道数据库表、后台页面和公开解析 API。
6. 联系页仍有假 400 电话和 `qrCode: null` 占位。
7. 二维码图片上传 API 当前只支持 `store`。

## 10. 验收标准

- [ ] 全站使用一个统一咨询组件和同一套解析规则。
- [ ] 总部可在后台替换、启用和停用二维码。
- [ ] 产品、项目、文章和门店能传入咨询上下文。
- [ ] 无有效二维码时不会显示破图或占位账号。
- [ ] 门店独立渠道不可用时能够回退总部渠道。
- [ ] `contact_click` 数据能区分渠道、页面来源和关联实体。
- [ ] 不采集个人微信号、手机号或聊天内容。
- [ ] 1440、768、390 三视口可扫码、可关闭、可降级。

- CRM 客户档案。
- 聊天记录同步。
- 自动欢迎语、群聊和朋友圈运营。
