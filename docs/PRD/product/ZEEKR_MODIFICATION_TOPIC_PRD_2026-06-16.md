## 2. 背景

蓝辉轻改正在从汽车膜、车衣、改色膜延展到新能源车型轻改装。极氪用户群体通常关注高端新能源座舱质感、内饰保护、尾箱收纳、防护配件、地板/脚垫、方向盘与踏板等细分升级项目。

本专题用于把极氪车系相关产品集中展示,让用户按车型快速了解可升级款式,并形成到店沟通或企业微信咨询意图。专题页必须服务品牌可信度,而不是简单堆叠 Excel 图片。

### 2.1 项目内图片规格现状(本版新增背景)

本版调研发现,项目内其他专题的产品图在像素和宽高比上**未做统一约束**,存在多源拼凑问题:

| --- | --- | ---: | ---: | --- |
| 问界 | `wenjie/preview.png` | 1100 × 1552 | 11:15.5(竖图) | 与产品图比例割裂 |
| 小米 | `xiaomi/su7/su7-01-front-bumper.png` | 2523 × 1661 | 1.52:1 | 比例失控 |
| 小米 | `xiaomi/su7/su7-02-pedals.png` | 781 × 490 | 1.59:1 | 比例失控 |
| 小米 | `xiaomi/su7/su7-03-seat-back-panel.png` | 654 × 691 | 1:1.06(近正方形) | 比例失控 |
| 小米 | `xiaomi/su7/su7-04-side-skirt.png` | 989 × 660 | 1.50:1 | 比例失控 |
| 小米 | `xiaomi/su7/su7-05-steering-wheel.png` | 800 × 634 | 1.26:1 | 比例失控 |

## 3. 用户与业务目标

### 3.1 用户角色

| 用户 | 需求 |
| --- | --- |
| 极氪车主 | 想按自己的车型快速看到可做哪些轻改项目,尤其是地板、尾箱、内饰保护、防护件等。 |

### 3.2 业务结果

本轮要建立一个极氪车型专题的前台展示闭环:

1. 用户在 `/product` 看到"极氪改装专题"入口。
2. 用户进入 `/product/zeekr`。
3. 用户按极氪 9X / 8X / 009 查看产品款式。
4. 用户通过卡片和表格理解产品名称、车型、分类、图片状态和卖点。
5. 用户通过全站统一咨询策略进入后续沟通;本 PRD 不单独规定专题私有咨询 CTA 文案。

## 8. 图片与资产策略(本版核心章节)

### 8.0 经验教训:为什么本版要固化图片规格

回顾 §2.1 现状盘点,项目内问界/小米专题的产品图**比例/像素均无约束**,直接后果:

- 卡片容器内图片大小不一,长宽比混合,布局跳动;
- 同一专题内不同产品图的视觉重量失衡,影响品牌专业感;
- 后续若用 Next/Image 优化(响应式 `sizes`、自动 srcset),无法稳定生成;
- 命名混乱(中文 / 英文拼写不一致如 `Zeeker009/1-borad.png` 中的 `borad` 应为 `board`),部署到 Linux/Vercel 时存在编码风险;
- **产品行 vs 图片文件不一一对应**(v1.0 PRD 假设 23 行=23 图,本版实地盘点发现 9X 14 图 / 8X 6 图 / 009 1 图,9X 行 8「挡泥板+内衬6件套」、行 9「双层脚垫」无对应 PNG,1 个 9X 文件「后备箱储物」与 PRD 行 15「后备箱储物盒」命名差异)——**这是 v1.0 隐式假设导致的盲点,本版显式拆出 `matched` / `pending-review` / `missing` 三态对冲**。

**本版极氪吸取以上教训,把图片规格从"软建议"升级为"项目级硬性约束"**:在物理文件层、数据类型层、构建期校验层、UI 容器层四个层面同时约束,缺一不可。

### 8.1 图片现状盘点

| 项 | 值 |
| --- | --- |
| 已就位文件总数 | 21(9X 14 + 8X 6 + 009 1) |
| 产品行总数 | 23(9X 16 + 8X 6 + 009 1) |
| 缺图产品行 | 2(均为 9X:行 8 挡泥板+内衬6件套、行 9 双层脚垫) |
| 已就位文件像素 | **1448 × 1086**(已天然一致) |
| 宽高比 | **4:3** |
| 格式 | PNG |
| 来源目录(迁移前) | `public/images/products/ZEEKR/{极氪9X,极氪8X,Zeeker009}/` |
| 目标目录(迁移后) | `public/images/products/zeekr/{9x,8x,009}/` |
| 文件命名 | 中文(迁移前) → ASCII slug(迁移后) |
| 状态分布 | 20 行 `matched`(图片已就位 + 命名对应产品行已核对,= 9X 13 + 8X 6 + 009 1)/ 1 行 `pending-review`(9X 行 15「后备箱储物.png」与 PRD §6.2 行 15「后备箱储物盒」命名差异,需业务核对)/ 2 行 `missing`(9X 行 8「挡泥板+内衬6件套」、行 9「双层脚垫」) |

> **说明**:上一版本 PRD 假设"23 行=23 图",本版实地盘点发现 21 张图、2 行缺图,1 个 9X 文件(`后备箱储物.png`)与 PRD §6.2 行 15「后备箱储物盒」是否同一产品需业务确认。本版显式拆出 `matched` / `pending-review` / `missing` 三态,避免"装作 23=23"的虚假完备。

### 8.2 物理规格硬性约束(项目级标准)

| 项 | 规格 | 验证方式 |
| --- | --- | --- |
| 像素 | 1448 × 1086 px(±0) | `sips -g pixelWidth -g pixelHeight` |
| 宽高比 | 4:3(强制) | width ÷ height = 1.333… |
| 格式 | PNG(透明背景优先)/ 退化 JPEG | 文件后缀 |
| 文件大小 | ≤ 3 MB(实测 21 张图 1051-2357 KB,平均 1675 KB) | `ls -l` |
| 命名 | ASCII slug,小写 + 短横线 | regex `^[a-z0-9-]+\.(png\|jpe?g)$` |
| 目录 | `public/images/products/zeekr/{9x,8x,009}/` | 路径前缀匹配 |
| 文件总数 | **21** 个 PNG(9X=14 + 8X=6 + 009=1),与数据层 `imageStatus="matched"` 计数一致 | glob 计数 |
| 缺图产品行 | 2 个(9X 行 8/9),UI 必须显式降级,**不**通过校验脚本阻断 build | 数据层校验 |
| 备选 slug 后缀 | `-pc`(件套)、`-cover`(套子)、`-guard`(防护) | 词表约束 |

> **本规格直接采用极氪 21 张已就位图片的实际像素(1448×1086)作为统一基准,不作另行挑选。** 这是因为 21 张图已经天然一致,把它固化为标准比"重新选定一个标准"成本低、风险小。

### 8.4 前端展示规格(UI 容器层)

```ts
export type ZeekrImageWidth = 1448;
export type ZeekrImageHeight = 1086;
export type ZeekrImageAspectRatio = "4/3";

export type ZeekrProductImage = {
  publicPath: string;       // 必填;迁移完成后 23 条全部为非空
```

- 失败时输出文件路径、原因、修复建议
- 通过 `console.error` 输出失败信息,`process.exit(1)`
- 不输出彩色日志(保持 CI 日志干净)

**`package.json` 接入:**

```json

  "scripts": {
```

```ts
  aspectRatio: ZeekrImageAspectRatio | null;

export interface ZeekrProduct {
  id: string;                        // zeekr-{modelLower}-{order},例 zeekr-9x-01
```

- 读取 Next.js 16 相关文档后再改 App Router 代码。
- 不在页面中引用微信缓存路径或本机绝对路径。
- 不在 JSX 中复制 23 个产品卡片硬编码,必须通过结构化数据 `map` 渲染。
- 不使用 `any`。
- 图片使用项目现有图片策略和 Next/Image 组件。
- 保持 Tailwind v4 与当前产品页风格一致(暗色主题、orange-500/400 主色)。

- 不破坏现有产品中心、门店页、新闻页、小米专题、问界专题和地板专题。
- 不引入新 UI 库。
- 不把整个页面变成 Client Component,除非车型 tabs 或埋点确实需要小型客户端组件。
- **所有 zeekr 卡片必须使用与 wenjie 相同的 `aspect-[4/3]` 容器与 `object-contain` 策略,禁止 `object-cover`(Hero 预览图除外)。**
- 完成后运行 `npm run lint`、`npm run typecheck`、`npm run verify:zeekr-images`、`npm run build`。

## 17. 风险与处理

| 风险 | 处理 |
| --- | --- |
| Excel 图片无法直接读取 | 实现前做图片提取脚本或人工导出,生成 manifest 后再接页面。 |
| 产品行 23 个但图片签名 24 个以上 | 不按顺序自动绑定图片,必须人工复核产品行与图片映射。 |
| 图片嵌入在微信缓存路径的 `.xls` 中 | 实现时复制到 `public`,不得引用源路径。 |
| 中文文件名影响部署 | 复制到 `public` 时改为 ASCII slug 文件名并保留映射。 |
| 车型名称可能需业务确认 | 架构师在执行前确认 `极氪9X`、`极氪8X` 是否为官网对外展示名称。 |
| 用户误认为官方授权 | 页面增加适配说明,避免"官方/授权/原厂"等措辞。 |
| 图片尺寸差异较大(本版不存在,作为预案) | 前端使用固定图片容器和 `object-fit` 策略。 |
| 专题页范围膨胀 | 本轮只做展示专题,不做报价、下单、库存、后台、3D 或单品详情。 |
| **图片规格失控(本版新增,关键风险)** | **本版建立 4 层约束(§8.2 物理 / §8.4 UI / §8.5 类型 / §8.6 CI)对冲。任一层失守均会导致布局跳动或规格漂移。** |
| **`/product` 入口顺序变化导致回归** | 在 `<XiaomiTopicBanner />`、`<WenjieTopicBanner />`、`<FlooringTopicBanner />` 之后追加 `<ZeekrTopicBanner />`,不动既有顺序。 |
| **校验脚本误报** | 脚本用 `image-size` 纯 Node 读取,不依赖系统命令;本地 macOS 与 Vercel Linux 行为一致。 |
| **校验脚本被绕过** | 接入 `npm run check`(`lint && typecheck && verify && build`),所有 CI 与本地预提交均会跑。 |
| **`Zeeker009` 拼写错误目录被遗漏** | §8.3.3 显式列迁移清单,实施时 grep 校验旧目录已删除。 |
| **产品行 vs 图片文件不一一对应(本版新增)** | 9X 行 8/9 无对应 PNG,1 个 9X 文件名「后备箱储物」与 PRD 行 15「后备箱储物盒」命名差异。本版用 `matched` / `pending-review` / `missing` 三态显式拆解,业务补图后由架构师在数据层 `imageStatus` 字段逐项标记。 |
| **缺图产品误配其他图(关键 UX 风险)** | 9X 行 8/9 缺图时,UI **禁止**用 8X 的「挡泥板+内衬6件套」图或地板专题的「脚垫」图填充;必须按 §8.4 缺图降级容器渲染。 |
