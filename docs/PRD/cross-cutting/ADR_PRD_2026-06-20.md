## 附录 A: 5 个 ADR 详细内容草案(本批补完,直接拷到 `docs/ADR/`)

### ADR-001: SSG 优先 + ISR 混合(为什么 `/admin` 用 force-dynamic)

**背景**:Next.js 16 App Router 提供 SSG / SSR / ISR / CSR 4 种渲染策略。公开站含 21 路由,CMS 含 10 路由,若全部用 SSR 则失去预渲染性能优势,若全部用 SSG 则 CMS 数据陈旧。

**评估的选项**:
| 选项 | 优点 | 缺点 |
|---|---|---|
| A. 全部 SSG | 性能最优 | CMS 数据陈旧,需手动重建 |
| B. 全部 SSR(`force-dynamic`) | 数据实时 | 首屏 6s+,LCP 全超标 |
| C. **混合:公开 SSG/ISR,CMS force-dynamic** | 性能 + 实时兼得 | 增加心智负担 |

**决策**:采用 C。`/admin/(dashboard)/*` 全部 `export const dynamic = "force-dynamic"`;公开站 `/agent/[slug]` 等动态路由 `generateStaticParams()` + `revalidate = 60` ISR;静态路由无 revalidate。

**后果**:
- ✅ 公开站 LCP < 2.5s(`/brand` 2.0s、`/news` 1.6s)
- ✅ CMS 数据每次请求重新查,无陈旧
- ⚠️ 新增路由时必须显式声明 `dynamic`(默认行为可能变化)

**不可变项**:
- 公开站 `/agent/store/[id]` 等**已上线路由**必须保留 ISR 60s
- `/admin/(dashboard)/*` 必须保留 `force-dynamic`

### ADR-002: 本地存储 vs 阿里云 OSS

**背景**:图片上传 API `/api/upload` 当前写入 `public/images/stores/<id>.webp`(本地存储);`ali-oss` 已安装但未 wire up;`.env.example` 含 OSS env vars(占位)。

**评估的选项**:
| 选项 | 优点 | 缺点 |
|---|---|---|
| A. **本地存储**(当前) | 零依赖、零成本、与 Next/Image 完美集成 | 容器删除则图片丢失,无 CDN 加速 |
| B. 阿里云 OSS + 图片样式 | CDN 加速、容量大 | 需额外配置、AUTH 风险、上传链路复杂 |
| C. 腾讯云 COS + 数据万象 | 同 B | 切换云厂商成本 |

**决策**:采用 A(本地存储),暂不上 OSS。

**理由**:
1. 当前 1 家真实门店 + 8 张证书,本地 `public/images/` 完全够用
2. 容器部署时 `public/` 已通过 `COPY --from=builder /app/public ./public` 持久化到镜像
3. 图片管理目前仅 admin 写、不公开上传,攻击面小
4. `ali-oss` 依赖保留(locked by package-lock),待门店数 ≥ 50 再评估迁移

**不可变项**:
- 上传 API `/api/upload` 必须 admin 角色 + sharp q80 + 文件名 `<storeId>.webp`
- 不要删除 `ali-oss` 依赖(已在 lock)

**何时复审**:门店数 ≥ 50 或月上传量 ≥ 1GB 时启动迁移评估。

### ADR-003: Prisma 7 + Driver Adapter(为什么不用 Prisma 6 旧 query API)

**背景**:Prisma 7 引入 Driver Adapter 机制,本项目用 `@prisma/adapter-pg` 直连 `pg` 驱动。Prisma 7 错误形态变更为 `meta.driverAdapterError.cause`,与 Prisma 6 的 `meta.target` 不兼容。

**评估的选项**:
| 选项 | 优点 | 缺点 |
|---|---|---|
| A. **Prisma 7 + adapter-pg**(当前) | 官方推荐、性能更好、可直连 pg pool | 错误形态变化、社区文档少 |
| B. Prisma 6 + 传统 query API | 文档多、错误形态熟悉 | 已停止维护,Prisma 团队主线在 7+ |
| C. Drizzle ORM | 类型推断更强 | 生态小、与 NextAuth adapter 集成度差 |

**决策**:采用 A。

**理由**:
1. Prisma 7 是 `prisma@^7.8.0` 当前版本,锁在 package.json
2. Driver Adapter 性能优于传统 query(实测 `findUnique` ~30%)
3. 项目 0 业务代码依赖 `meta.target`,错误形态迁移成本低

**不可变项**:
- 严禁 `npm install prisma@6`(即使短期解决某个 bug)
- Prisma 错误处理统一走 `meta.driverAdapterError.cause`(见 `src/lib/prisma-error.ts`)

**何时复审**:Prisma 8 发布时评估;或当 adapter-pg 与 pg 16+ 出现兼容问题时。

### ADR-004: NextAuth v5 beta + JWT(为什么不用 DB session)

**背景**:NextAuth v5 默认 JWT 策略,本项目用 `CredentialsProvider` + `bcryptjs` 10 rounds + JWT,无 `Session` 表。

**评估的选项**:
| 选项 | 优点 | 缺点 |
|---|---|---|
| A. **NextAuth v5 beta + JWT**(当前) | 零 DB session 查询、JWT 内置 role、SSG 友好 | 无法服务端主动失效 JWT(除非黑名单) |
| B. NextAuth v4 + DB session | 文档成熟、可服务端 kill session | v4 已 EOL、需多查 DB |
| C. 自建 JWT(脱离 NextAuth) | 完全控制 | 重造轮子、与 Next.js middleware 集成差 |

**决策**:采用 A。

**理由**:
1. 角色(角色 admin / editor)简单,JWT payload 即可
2. `src/app/admin/(dashboard)/layout.tsx` 用 `auth()` 守卫,JWT 校验 < 1ms
3. 无"主动踢出用户"需求(只 1 家门店 + 几个内部账号)
4. Prisma adapter 已装但未使用,作为升级到 OAuth(如企业微信登录)的预留

**不可变项**:
- `src/lib/auth.ts` 必须用 `next-auth@^5.0.0-beta.31` 配置
- role 字段必须写进 JWT(`jwt({token, user})` 回调)
- 不要引入 DB session 表(即使以后加 OAuth 也建议保持 JWT)

**何时复审**:出现"立即禁用某账号"业务需求时,改为 JWT 黑名单 + 短期 expiry。

### ADR-005: 主题专项 5 组件结构

**背景**:已实施 wenjie / xiaomi / zeekr 三个主题页,均采用统一 5 组件结构。下一个主题(如 BYD)若不遵守,会风格漂移。

**评估的选项**:
| 选项 | 优点 | 缺点 |
|---|---|---|
| A. **每主题 5 组件**:AnchorNav / ProductCard / ProductGrid / ProductTable / TopicBanner | 复用模式、CI 可校验、字面量类型防规格漂移 | 早期工作量略大 |
| B. 自由发挥,后整合 | 单主题快 | 跨主题难对比、CI 无法校验 |
| C. 用 1 个万能组件 | 看起来 DRY | 配置项爆炸、维护成本高 |

**决策**:采用 A。

**理由**:
1. ZEEKR build 验证:5 组件 + 字面量类型(`Width=1448 / Height=1086 / Ratio="4/3"`)让 CI 可校验
2. 同一主题内:AnchorNav 锚点跳转 + ProductGrid 桌面网格 + ProductTable 移动表格 + ProductCard 3 态(`matched / pending-review / missing`)
3. 已用 `scripts/verify-zeekr-images.mjs` 验证(链入 `npm run check`)

**不可变项**:
- 每个新主题必须输出 `src/lib/<topic>-products.ts`(literal types) + 5 组件 + `scripts/verify-<topic>-images.mjs`
- 主题 banner 配色:xiaomi=orange、wenjie=cyan、zeekr=orange、flooring=amber(已有);新主题选色前必须查 README 避免冲突
- 图片容器统一:`aspect-[4/3] + object-contain + Next/Image sizes`

**何时复审**:出现"主题页 = 4 个车型"以下需求时(组件冗余);或主题数 ≥ 10 时考虑配置化(以主题为 key 的注册表)。

## 附录 B: scripts/verify-adr.mjs 草案

```js
#!/usr/bin/env node
// 验证 docs/ADR/ 下的 ADR 文件符合规范
// 1. NNNN 编号 4 位 0-pad,单调递增
// 2. 每份含 8 个必填节
// 3. Status 必须在 5 个合法值之内
import fs from "node:fs";
import path from "node:path";

const ADR_DIR = path.join(process.cwd(), "docs/ADR");
const VALID_STATUSES = ["Proposed", "Accepted", "Deprecated", "Superseded"];
const REQUIRED_SECTIONS = [
  "背景", // 也接受英文 "Context"
  "评估的选项",
  "决策",
  "后果",
  "验证",
  "参考",
];

if (!fs.existsSync(ADR_DIR)) {
  console.error("[verify-adr] docs/ADR/ 目录不存在");
  process.exit(1);
}

const files = fs
  .readdirSync(ADR_DIR)
  .filter((f) => /^\d{4}-.+\.md$/.test(f))
  .sort();

let prev = 0;
let errors = 0;
for (const f of files) {
  const num = parseInt(f.slice(0, 4), 10);
  if (num !== prev + 1) {
    console.error(`[verify-adr] ${f} 编号不连续(应为 ${String(prev + 1).padStart(4, "0")})`);
    errors++;
  }
  prev = num;

  const content = fs.readFileSync(path.join(ADR_DIR, f), "utf8");
  const statusMatch = content.match(/^>\s*状态:\s*(.+)$/m);
  if (!statusMatch || !VALID_STATUSES.includes(statusMatch[1].trim())) {
    console.error(`[verify-adr] ${f} 状态非法: ${statusMatch?.[1]}`);
    errors++;
  }
  for (const sec of REQUIRED_SECTIONS) {
    if (!content.includes(`## ${sec}`)) {
      console.error(`[verify-adr] ${f} 缺少 ## ${sec}`);
      errors++;
    }
  }
}

if (errors > 0) {
  console.error(`[verify-adr] 共 ${errors} 个错误`);
  process.exit(1);
}
console.log(`[verify-adr] ${files.length} 份 ADR 全部合规`);
```
